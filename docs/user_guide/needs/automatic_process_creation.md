# Automatic creation of multidisciplinary simulation processes { #needs-automatic-creation-of-multidisciplinary-simulation-processes }

## The problem { #needs-the-problem }

Once simulation tools are wrapped as disciplines,
the next challenge is to **assemble them into a process**.
In a traditional approach,
the engineer must manually define
which discipline runs first,
how data flows between them,
and how coupled disciplines are resolved.
For a handful of disciplines this is manageable,
but as the number grows---10, 20, 50 disciplines---the task becomes
overwhelming and error-prone.

Worse,
every time a discipline is added, removed, or its interface changes,
the entire process may need to be redesigned.
This manual effort
represents a significant development and maintenance cost.

## GEMSEO's answer: automatic process generation { #needs-gemseos-answer-automatic-process-generation }

GEMSEO **automatically generates the simulation process**
from the set of disciplines and a chosen strategy called an **MDO formulation**.

The mechanism works in three steps:

1. **Graph analysis**:
   GEMSEO analyzes the inputs and outputs of all disciplines
   to build a **coupling graph**.
   This graph reveals which disciplines are connected
   and which form **strongly coupled groups**
   (i.e. groups with circular dependencies).

2. **Process construction**:
   Based on the chosen MDO formulation (e.g. MDF, IDF, BiLevel),
   GEMSEO automatically builds the execution sequence.
   Coupled groups are resolved by inserting appropriate
   multidisciplinary analysis (MDA) solvers.
   Uncoupled disciplines are chained sequentially.

3. **Execution**:
   The process is driven by an optimization algorithm or a DOE,
   which calls the disciplines in the right order,
   passing the right data at each step.

The key benefits of this approach are:

- **Reduced development cost**: the engineer describes *what* the problem is, not *how* to solve it.
- **Reduced maintenance cost**: adding or modifying a discipline only requires updating its interface; the process adapts automatically.
- **Deep reconfiguration**: switching from one MDO formulation to another is a one-line change.
